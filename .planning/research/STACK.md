# Technology Stack

**Project:** Astra RM Robot Platform
**Researched:** 2026-03-30

## Recommended Stack

The repo should stay a **C11 + CMake + Python + STM32CubeMX** platform. The right move for v1 is not to replace the stack, but to add a **host-native verification lane** beside the existing hardware and SITL lanes so control logic, message contracts, and fake-link behavior can be tested before `generate` or on-robot work.

### Core Framework
| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| C11 | Existing repo standard | Runtime, control, device, and contract code | Keep. The active code under `robot_platform/runtime/` is already C-first and shared across hardware and SITL via `robot_platform/CMakeLists.txt`. Confidence: HIGH |
| CMake + CTest | CMake 3.22+ | Single build graph for hardware, SITL, and new host tests | Add `enable_testing()` and host test targets inside the existing CMake graph instead of introducing a second test build system. This preserves the current `generate -> build -> sim -> test` flow in `robot_platform/tools/platform_cli/main.py`. Confidence: HIGH |
| Python 3 | Existing repo runtime | Orchestration CLI, smoke runner, report validation | Keep Python on the tooling side only. The current CLI and SITL harness already live in `robot_platform/tools/` and `robot_platform/sim/`, so Python should remain the outer orchestration layer, not the control-logic layer. Confidence: HIGH |

### Verification and Test
| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Unity | 2.x, vendored | C unit-test assertions for host-side runtime tests | Best fit for small embedded C modules such as `message_center`, control contracts, and mapping logic. Lightweight, C-native, and easy to compile inside the current CMake host toolchain. Confidence: MEDIUM |
| FFF (Fake Function Framework) | vendored snapshot | Function fakes for BSP/device seams and fake-link validation | Prefer fakes over heavy autogenerated mocks for this stage. The repo needs to fake IMU, remote, actuator, and link behavior around code in `robot_platform/runtime/device/` and `robot_platform/runtime/control/`. FFF is a better fit than a separate mocking toolchain. Confidence: MEDIUM |
| AddressSanitizer + UndefinedBehaviorSanitizer | GCC/Clang host builds | Memory and UB detection in host-native tests and SITL | Keep the existing sanitizer direction signaled by `PLATFORM_SIM_ASAN` in `robot_platform/CMakeLists.txt`, but extend it to all host verification targets. This is especially important for the payload-size risk in `robot_platform/runtime/module/message_center/`. Confidence: HIGH |
| gcovr | 7.x class tool | Coverage reporting for host C tests | Add coverage only for host-native C tests. This gives useful signal on control-path gaps without pretending hardware builds should carry coverage instrumentation. Confidence: MEDIUM |
| Python `unittest` | Existing | CLI, report, and sim-summary regression tests | Keep the current Python test surface in `robot_platform/tools/platform_cli/tests/` and `robot_platform/sim/tests/`. Expand it only for orchestration/report behavior, not for core control logic. Confidence: HIGH |

### Build and Toolchain
| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|

| GCC on Linux | Existing host toolchain | SITL and host-native C tests | Keep and standardize on the existing Linux host path in `robot_platform/cmake/toolchains/linux-gcc.cmake` for TDD, sanitizers, and coverage. Confidence: HIGH |

| STM32CubeMX 6.17.0 | Current repo expectation | Board code generation from `Astra_RM2025_Balance/Chassis/CtrlBoard-H7_IMU.ioc` | Keep, but treat it as a staged artifact generator, not part of the inner TDD loop. `generate` should run after host tests pass, not before every control change. Confidence: HIGH |

## Repo-Fit Decisions

### Keep
- Keep the single platform entrypoint in `robot_platform/tools/platform_cli/main.py`.
- Keep the shared C runtime structure in `robot_platform/runtime/` and the current cross-target CMake topology in `robot_platform/CMakeLists.txt`.
- Keep `STM32CubeMX` and the current `.ioc` source of truth in `Astra_RM2025_Balance/Chassis/CtrlBoard-H7_IMU.ioc`.
- Keep SITL as an integration and smoke lane, not the primary unit-test lane.

### Add
- Add a new **host test target family** under the existing CMake graph for pure-C tests of modules in `robot_platform/runtime/`.
- Add lightweight C fakes around BSP/device seams so fake-link validation can exercise topic transport, actuator mapping, stale-link handling, and guard logic without the SITL process boundary.
- Add sanitizer-enabled host presets or CLI modes for `unit`, `contract`, and `fake_link` stages.
- Add coverage reporting for host C tests only, with thresholds on the safety-critical runtime modules first: `message_center`, actuator mapping, control contracts, and startup/observe paths.
- Add a freshness/check step that fails fast when generated board artifacts and the `.ioc` source drift.

### Avoid
- Avoid Ceedling/CMock as the primary path. It would split the repo into a second build/test workflow with Ruby-specific conventions that do not match the existing CMake-driven platform.
- Avoid GoogleTest/Catch2-first testing. The runtime is C, and a C++-centric harness would add wrapping cost without solving the actual fake-link problem.
- Avoid pushing more validation into Python when the real gap is untested C runtime logic.
- Avoid treating SITL smoke success as proof of control correctness. The current repo context already shows the SITL path is partially stubbed and the smoke layer mostly validates report plumbing.
- Avoid any stack move toward ROS 2, Gazebo, or a larger middleware migration for v1. That is scope expansion, not trust-building.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| C unit testing | Unity + FFF inside CMake | Ceedling + CMock | Good embedded tooling, but it creates a parallel build/test system and increases workflow friction in this brownfield repo. Confidence: MEDIUM |
| Integration verification | Existing SITL + targeted fake-link tests | SITL-only validation | The current SITL lane is not yet trustworthy enough to carry all pre-hardware validation by itself. Confidence: HIGH |
| Python testing | Keep `unittest` for tooling tests | Migrate immediately to `pytest` | Possible later, but not the highest-value change for v1. The current bottleneck is missing C runtime tests, not Python test ergonomics. Confidence: MEDIUM |
| Build orchestration | One CMake graph with staged targets | Separate bespoke scripts per lane | The repo already has a platform CLI and CMake graph; fragmenting orchestration would make reliability worse. Confidence: HIGH |

## Recommended Strategy for Current Stage

For this project stage, the most appropriate stack strategy is:

1. Keep the current embedded platform stack intact.
2. Add host-native C testing inside the existing CMake graph.
3. Use fakes to validate data-link and device/control contracts before SITL.
4. Use SITL as a higher-level smoke lane after host tests, not instead of them.
5. Keep hardware generation and flashing outside the fast inner loop.

This gives the repo a trustworthy staged path:

`C unit/contract tests -> fake-link tests -> SITL smoke -> firmware build/generate -> constrained hardware bring-up`

## Confidence Notes

- **HIGH:** Keeping C11, CMake, Python orchestration, GCC toolchains, CubeMX, and SITL as the base stack. This is directly supported by `robot_platform/CMakeLists.txt`, `robot_platform/tools/platform_cli/main.py`, `robot_platform/sim/core/runner.py`, and `robot_platform/docs/wsl_environment_setup.md`.
- **MEDIUM:** Recommending Unity + FFF specifically. This is a repo-fit recommendation based on embedded C brownfield patterns and the code’s current seams, but it was not validated with network docs in this session.
- **HIGH:** Avoiding a larger framework migration in v1. The repo’s own concerns show that trust gaps are in runtime correctness and staged validation, not in missing middleware.
