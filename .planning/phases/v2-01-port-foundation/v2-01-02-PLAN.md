    **Step 1: Create `robot_platform/runtime/bsp/ports.h`**

    This is the canonical BSP port interface. It declares simple function signatures — no vtables, no structs, no device handles. The types it uses come from the existing contract headers.

    ```c
    #ifndef PLATFORM_BSP_PORTS_H
    #define PLATFORM_BSP_PORTS_H

    #include "../control/contracts/device_input.h"
    #include "../control/contracts/device_feedback.h"
    #include "../control/contracts/actuator_command.h"
    #include "../device/device_types.h"

    /*
     * BSP port interface — link-time polymorphism.
     *
     * Each target (HW, SITL, host-test) provides exactly one .c file
     * implementing these functions. CMake selects which .c to link.
     *
     * These ports coexist with the current device_layer path.
     * Task code does NOT call these yet — that migration is Phase 2.
     */

    platform_device_result_t platform_imu_read(platform_imu_sample_t *out);
    platform_device_result_t platform_remote_read(platform_rc_input_t *out);
    platform_device_result_t platform_motor_read_feedback(platform_device_feedback_t *out);
    platform_device_result_t platform_motor_write_command(const platform_motor_command_set_t *cmd);

    #endif
    ```

    Use relative includes from `bsp/` to `control/contracts/` and `device/`. Check that the include paths resolve correctly given the CMake include directories already configured for each target.

    NOTE: The include path `../control/contracts/device_input.h` assumes `bsp/` is at the same level as `control/` inside `runtime/`. Verify this matches the directory structure. If CMake include directories already provide `PLATFORM_CONTROL_CONTRACT_DIR`, you may use `#include "device_input.h"` instead — but prefer the relative path for explicitness since this header will be included from multiple targets with different include configurations.

    **Step 2: Create `robot_platform/runtime/bsp/boards/stm32h7_ctrl_board/bsp_ports_hw.c`**

    HW implementation. For now, these are thin delegation stubs that call through the existing device_layer default API. This ensures the port functions exist and link, while the actual HW driver binding stays in device_layer until Phase 2 migrates callers.

    ```c
    #include "../../ports.h"
    #include "../../../device/device_layer.h"

    platform_device_result_t platform_imu_read(platform_imu_sample_t *out)
    {
        return platform_device_read_default_imu(out);
    }

    platform_device_result_t platform_remote_read(platform_rc_input_t *out)
    {
        return platform_device_read_default_remote(out);
    }

    platform_device_result_t platform_motor_read_feedback(platform_device_feedback_t *out)
    {
        return platform_device_read_default_feedback(out);
    }

    platform_device_result_t platform_motor_write_command(const platform_motor_command_set_t *cmd)
    {
        /* Wrap the command set in a device command for the current device_layer API */
        platform_device_command_t device_cmd = {0};
        device_cmd.motors = *cmd;
        return platform_device_write_default_command(&device_cmd);
    }
    ```

    NOTE on `platform_motor_write_command`: The current `platform_device_write_default_command` takes `const platform_device_command_t *` which wraps the motor command set. After Plan 01 unifies the command types, `platform_device_command_t` becomes an alias for `platform_actuator_command_t`. Adjust the delegation accordingly — if Plan 01 has already run, the wrapping may need to construct a `platform_actuator_command_t` or the write function may accept `platform_motor_command_set_t` directly. Read the current state of `device_layer.h` before implementing.
