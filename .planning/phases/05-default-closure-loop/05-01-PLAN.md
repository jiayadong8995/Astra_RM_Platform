  <name>Task 1: Add validate command with sequential stage pipeline and closure artifact</name>
  <files>robot_platform/tools/platform_cli/main.py</files>
  <action>
Add the `validate` command to the existing CLI. This involves:

1. Add `_parse_validate_args(args: list[str]) -> tuple[str, Path]` that accepts `--project` (default "balance_chassis") and `--report` (default "build/closure_reports/closure_balance_chassis.json"). Follow the same arg-parsing pattern as `_parse_verify_phase1_args`.

2. Add `_write_closure_report(report_path: Path, stages: list[dict], project: str, failure_stage: str | None, failure_reason: str | None, hw_elf_status: str | None) -> None` helper that writes the closure JSON artifact:
```python
{
    "closure_version": 1,
    "project": project,
    "overall_status": "passed" | "failed",
    "failure_stage": None | "stage_name",
    "failure_reason": None | "reason_string",
    "hw_elf_attempted": bool,
    "hw_elf_status": None | "passed" | "failed" | "skipped",
    "stages": stages,
    "timestamp": "ISO-8601 string from datetime.datetime.now(datetime.timezone.utc).isoformat()"
}
```
Use `report_path.parent.mkdir(parents=True, exist_ok=True)` and `report_path.write_text(json.dumps(..., indent=2) + "\n")`. Import `datetime` at the top of the file.

3. Add `_run_validate(project: str, report_path: Path) -> int` that sequences these stages using the existing `_run_stage` helper, with early-exit after each:

   Stage 1 - `build_sitl`: `_run_stage("build_sitl", lambda: _build_sitl(profile.sitl_target))` where profile comes from `get_project_profile(project)`. If profile is None, print error and return 2.

   Stage 2 - `host_tests`: `_run_stage("host_tests", lambda: _run_host_ctest([...all PHASE2_CASES targets...], "...|..." regex joining all PHASE2_CASES regexes))`. This runs all host C safety tests. Build the targets list and regex from PHASE2_CASES dict keys.

   Stage 3 - `python_tests`: `_run_stage("python_tests", lambda: _run_tests("sim"))`. This runs Python CLI + sim tests.

   Stage 4 - `smoke`: Run SITL smoke with `skip_build=True` since build_sitl already ran. Use the same smoke report loading and status checking pattern from `_run_verify_phase1` (lines 477-497). The stage dict should include `smoke_report` path and `runtime_output_observation_count` like phase3 does.

   Stage 5 - `verify_phase3`: Run `_run_verify_phase3(project, Path("build/verification_reports/phase3_balance_chassis.json"), None)` as a stage. This internally builds SITL again but that's fast (already built). Alternatively, to avoid redundancy, call `_run_stage("verify_phase3", lambda: _run_verify_phase3(project, repo_root / "build" / "verification_reports" / "phase3_balance_chassis.json", None))`.

   After all 5 validation stages pass, attempt hw_elf as best-effort:
   Stage 6 (optional) - `build_hw_elf`: Attempt `_build_hw_seed("balance_chassis_hw_seed.elf")` wrapped in `_run_stage`. If it fails, record the failure in the closure artifact but keep `overall_status` as "passed" since all validation gates passed. Skip the freshness gate for now (it will fail without a manifest) — wrap the attempt in a try and record "skipped" if freshness check fails. Specifically: run `_require_generated_artifact_freshness()` first; if it returns non-zero, set hw_elf_status to "skipped" with reason "freshness_gate_failed"; otherwise attempt the build.

   For each stage 1-5, on failure: call `_write_closure_report` with the accumulated stages, the failing stage name, and return 1.
   On full success: call `_write_closure_report` with all stages, no failure, and return 0.

   Print a one-line summary after writing the report: `closure summary: project={project} overall_status={status} failure_stage={failure_stage}`
   Print the report path: `closure report: {resolved_report_path}`

4. Add routing in `main()`: After the `verify` block, add:
```python
if cmd == "validate":
    try:
        project, report_path = _parse_validate_args(args[1:])
    except ValueError as exc:
        print(str(exc), file=sys.stderr)
        return 2
    return _run_validate(project, report_path)
```

5. Add "validate" to the HELP dict: `"validate": "Run full closure loop: build, test, smoke, verify, firmware."`.

6. Update the help text in `main()` where it prints supported commands to include "validate".

Important implementation notes:
- Reuse `_run_stage` for all stages — do NOT create a new timing/status wrapper.
- The `_run_verify_phase3` call inside validate will do its own SITL build internally. To avoid this redundant build, consider calling the phase3 verification with the SITL already built. Looking at `_run_verify_phase3`, it calls `_build_sitl` directly (not via `_run_stage`). The simplest approach: let it rebuild (it's fast, cmake will no-op). Don't try to refactor `_run_verify_phase3` to accept a skip_build flag — that's scope creep.
- Resolve report_path relative to repo_root if not absolute, same pattern as verify commands.
- The smoke stage inside validate is separate from the verify_phase3 smoke. validate's smoke is a standalone gate; verify_phase3 runs its own smoke internally for the phase3 case matrix. This is intentional — verify_phase3 needs its own smoke to produce the phase3 artifact.
