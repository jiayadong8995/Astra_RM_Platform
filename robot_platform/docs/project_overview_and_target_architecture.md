# Project Overview And Target Architecture

这份文档只回答四个问题：

1. 这个项目现在到底是什么
2. 整个机器人的运行主链是什么
3. 平台应该分成哪几层
4. 当前项目具体偏在哪里，后续应怎样摆正

它的目的不是替代详细设计，而是先把整个项目的总览地图立起来。

## 1. 项目定义

`Astra_RM_Platform` 当前应被理解为一个面向 RoboMaster 控制场景的嵌入式平台项目。

它不是：

- 单纯继续维护一份 `CubeMX + Keil + legacy task` 固件
- 单纯把旧代码搬到新目录
- 单纯围绕某一个底盘工程做局部重构

它要做的事情是：

- 统一生成链
- 统一构建链
- 统一运行时分层
- 统一验证链

当前仓库中的三类资产分别承担不同职责：

1. `Astra_RM2025_Balance/`
   历史基线资产。
   保存原始行为、原始固件组织方式、`.ioc` 和驱动参考，不作为主开发入口。
2. `robot_platform/`
   当前主工程。
   平台架构定义与落地的主执行面。
3. `Platform_Refactor/`
   路线决策归档。
   用于记录为什么项目收敛到当前路线，不承担执行面职责。

## 2. 当前阶段目标

当前阶段的目标不是继续讨论“是否迁移”，而是把平台真正立住。

当前核心目标只有四个：

1. 让 `hw` 与 `sitl` 共用主控制链
2. 让新消息总线成为统一消息机制
3. 让硬件细节从业务层剥离
4. 让设备层、控制运行时层、业务装配层分开

当前非目标也要明确：

- 不做 `osal`
- 不做 `physics_sim`
- `replay` 暂不作为当前交付项
- 不做大范围算法重写
- 不同时展开多个机器人项目

## 3. 机器人运行主链

整个机器人的运行流程，应按下面这条主链理解：

1. 系统启动
2. 设备初始化
3. 原始输入采集
4. 状态估计与观测
5. 业务决策
6. 控制计算
7. 执行器下发
8. 反馈回流形成闭环

可以压缩成一句话：

`设备输入 -> 状态估计 -> 业务决策 -> 控制计算 -> 执行器下发 -> 反馈闭环`

### 3.1 系统启动

职责：

- 芯片启动
- 时钟、中断、外设初始化
- RTOS 启动
- 创建任务

这一步只回答：

系统能不能正常跑起来。

### 3.2 设备初始化与输入采集

职责：

- IMU、遥控器、电机、CAN、串口等开始工作
- 原始硬件数据进入系统
- 执行器具备命令接收能力

这一步只回答：

设备是否可用，原始输入是否到位。

### 3.3 状态估计与观测

职责：

- 把原始设备输入转成控制真正需要的状态
- 姿态、速度、腿长、腿角、轮速等状态在这里形成

这一步回答：

当前机器人真实状态是什么。

### 3.4 业务决策

职责：

- 根据遥控、模式和当前状态决定机器人要做什么
- 例如待机、站立、移动、跳跃、转向

这一步回答：

当前机器人意图是什么。

### 3.5 控制计算

职责：

- 根据目标状态和当前状态做控制计算
- 输出关节力矩、电流、速度等控制命令

这一步回答：

应该给执行器什么命令。

### 3.6 执行器下发与反馈回流

职责：

- 把控制结果翻译成设备命令
- 下发到电机和执行器
- 收集反馈，再送回状态估计和控制链

这一步保证：

系统闭环运行，而不是单向算控制量。

## 4. 平台目标分层

当前项目更合理的目标层次应当是：

```text
generated
-> bsp
-> device
-> runtime_service
-> module
-> app
-> sim
```

各层职责如下。

### 4.1 `generated`

职责：

- 承接 `CubeMX` 生成资产
- 保存启动代码、HAL 初始化、外设初始化

原则：

- 不承接人工业务逻辑

### 4.2 `bsp`

职责：

- 板级支持
- 具体驱动实现
- 总线、句柄、寄存器、芯片级访问

原则：

- 对上暴露能力
- 不直接承接业务决策

### 4.3 `device`

职责：

- 建立统一设备节点模型
- 对传感器、输入设备、执行器做统一抽象
- 隔离具体驱动差异

建议设备类型至少包括：

- `imu`
- `remote_control`
- `joint_actuator`
- `wheel_actuator`

原则：

- 上层只认设备语义
- 不认具体驱动实现和硬件句柄

### 4.4 `runtime_service`

职责：

- 把设备层与控制链真正接起来
- 承接输入整理、反馈采集、状态估计组织、执行器运行时、控制运行时 glue

建议至少拆成：

- `sensing`
- `observe`
- `actuator`
- `control_runtime`

这层是当前项目最缺的一层。

### 4.5 `module`

职责：

- 算法
- 控制器
- 消息总线
- 公共控制模块

原则：

- 尽量不反向依赖 app 和板级细节

### 4.6 `app`

职责：

- 业务装配
- 模式管理
- 任务编排
- 项目级入口

它只负责：

- 机器人要做什么
- 任务怎么组织
- 业务边界怎么暴露

它不应直接负责：

- 电机驱动
- 设备反馈采集
- 硬件句柄
- 大量控制运行态 glue

### 4.7 `sim`

职责：

- 平台公共验证框架
- 承接 `SITL`
- 后续承接可扩展 backend

原则：

- `sim core` 公共
- 每个机器人通过 profile/adapter 接入
- 未来 `physics`、`vision` 通过新增 backend 或 adapter 接入

## 5. 当前项目偏差点

当前项目主方向没有错，偏的是运行时分层。

### 5.1 偏差一：把 task 拆分当成平台分层

当前很多工作是在把 legacy task 拆成不同文件和目录。

这有价值，但它解决的是：

- 文件组织问题

没有真正解决：

- 稳定职责边界问题

### 5.2 偏差二：`app` 吃进了执行层

目前 `app` 中仍然存在：

- 执行器命令下发
- 执行器反馈采集
- 设备反馈到控制运行态的映射

这意味着 `app` 不只是业务装配层，而是已经越界。

### 5.3 偏差三：缺少 `device` 层

当前业务层和运行时层仍然容易直接碰：

- 电机结构体
- CAN 句柄
- 具体驱动函数
- 硬件状态细节

这说明设备抽象没有立住。

### 5.4 偏差四：缺少 `runtime_service` 层

当前项目里最应该存在、但还没真正独立出来的是：

- 输入整理
- 反馈采集
- actuator runtime
- control runtime glue

结果就是：

- `module` 太早接触运行态
- `app` 又吃进了执行细节

## 6. 当前需要摆正的方向

接下来不应继续用“哪里耦合重就拆哪里”的方式推进。

正确顺序应当是：

1. 先确认项目总览和目标层次
2. 再识别当前代码分别落在哪一层
3. 先补 `device`
4. 再补 `runtime_service`
5. 再把 `app` 瘦回业务装配层
6. 最后再把 legacy task 压成真正 shell

## 7. 当前阶段的一句话目标

当前阶段的平台目标可以压缩成一句话：

围绕 `balance_chassis`，先补齐 `device` 和 `runtime_service` 两层，收紧 `module / app / bsp` 的真实边界，并让 `hw + sitl` 在同一条主控制链上稳定闭环。
