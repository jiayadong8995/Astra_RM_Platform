# Runtime Architecture Task List

这份文档定义当前阶段架构工作的执行清单。

目标不是继续围绕遗产代码做搬运，而是先固定未来重构和迁移必须遵守的边界、契约和顺序。

---

## 1. 当前阶段目标

当前阶段只做三件事：

1. 固定目标架构
2. 固定正式数据契约
3. 固定迁移顺序与禁区

不在这一阶段做的事情：

- 不继续按 legacy task 组织架构
- 不继续扩散 runtime glue 层
- 不先写大量实现代码
- 不先优化遗产代码细节

---

## 2. 任务清单

## T1. 固定顶层分层

目标：

- 正式确认 `generated / bsp / device / control / app / sim` 为目标主架构

交付：

- 架构蓝图文档
- 各层职责说明
- 分层边界禁区说明

完成标准：

- 每一层都能回答“负责什么”和“不负责什么”
- 不再用 legacy task 名称当架构单位

## T2. 固定主数据流

目标：

- 正式确认控制主链数据流

主链：

```text
DeviceInput
  -> RobotState
  -> RobotIntent
  -> ActuatorCommand
  -> DeviceCommand
  -> Backend
  -> DeviceFeedback
  -> RobotState
```

交付：

- 数据流图
- 核心边界对象列表

完成标准：

- 明确 `ActuatorCommand` 与 `DeviceCommand` 的区别
- 明确主控制链不采用“全量 pub-sub 化”

## T3. 固定正式契约对象

目标：

- 定义后续所有迁移必须遵守的正式数据对象

对象范围：

- `DeviceInput`
- `RobotState`
- `RobotIntent`
- `ActuatorCommand`
- `DeviceCommand`
- `DeviceFeedback`

交付：

- 契约草案文档
- 字段分层建议
- 每个对象的使用边界

完成标准：

- 上层不再直接依赖散落状态结构
- 新实现优先围绕正式契约展开

## T4. 固定通信原则

目标：

- 明确哪些边界使用直接接口，哪些边界允许广播机制

原则：

- 主控制链：直接接口 + 明确结构体
- 广播链：轻量 pub-sub
- 请求式操作：显式调用或 request-response

交付：

- 通信矩阵

完成标准：

- 不再把“使用消息总线”误当成“架构完成”

## T5. 固定能力边界

目标：

- 用“能力域”而不是“task 文件名”拆系统

能力域：

- 设备能力
- 状态形成能力
- 控制求解能力
- 执行映射能力
- 应用装配能力

交付：

- 能力边界表
- 层归属表

完成标准：

- 每类能力都能明确归属
- 不再把设备、控制、业务混在同一层

## T6. 固定迁移顺序

目标：

- 让后续重构按新契约和新分层推进

建议顺序：

1. 先立 contracts
2. 再立 `device`
3. 再立 `control/state`
4. 再立 `control/controllers`
5. 再立 `control/execution`
6. 最后收 `app`

交付：

- 迁移阶段清单
- 每阶段进入条件和退出条件

完成标准：

- 迁移按架构目标推进，而不是按旧文件大小推进

---

## 3. 下一步优先级

当前最优先的 3 个任务：

1. 完成 `T3`，写出正式契约对象草案
2. 完成 `T5`，写出能力边界表
3. 完成 `T6`，写出迁移顺序、进入条件和禁区

---

## 4. 迁移禁区

后续迁移中，默认禁止以下做法：

- 继续以 `INS_task`、`chassis_task`、`remote_task` 作为架构单位
- 继续把控制运行态塞回 `app`
- 继续把设备细节暴露到业务层
- 继续让 `ActuatorCommand` 直接等于底层发送参数
- 继续把消息总线当作主控制链唯一通信方式

---

## 5. 阶段完成判定

当前阶段完成的判定标准是：

- 目标架构稳定
- 正式契约稳定
- 能力边界稳定
- 迁移顺序稳定

满足这四条之后，才进入下一阶段的代码迁移。
